# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A static personal website ("Towards Wonder") for Elvira Wang — plain HTML/CSS/JS with no build tooling, no framework, and no runtime dependencies. Every page is hand-authored and linked via relative paths. `package.json` exists, but only for two Node dev scripts that prepare gallery images (Photographs, Drawings and Concerts alike) before deploy (see "Gallery architecture" → "Adding real images is automated" below) — it does not add a build step to the site itself.

## Commands

There is no build, lint, or test setup for the site itself. To preview changes, either:
- Double-click `index.html` to open it directly in a browser, or
- Serve the directory locally, e.g. `python3 -m http.server` from the project root, then visit `http://localhost:8000`.

To publish: drag the folder into Netlify Drop, or push to a GitHub Pages–enabled repo (per [README.txt](README.txt)).

If you've added photos to `images/gallery/{photographs,drawings,concerts}/`, run `npm install` once, then `npm run prepare-gallery` before deploying — see "Gallery architecture" for what this does. This is a manual step by design (the site itself has no build/server process, so nothing regenerates automatically on its own) — but `npm run watch-gallery` will run it automatically for you, in the background, every time a file changes under `images/gallery/`, for as long as you leave it running in a terminal. It does not make the *deployed* page auto-update (that still needs a commit + redeploy after you're done); it only keeps your local preview in sync while you're adding images, so you don't have to remember to re-run the command by hand each time.

## Architecture

- `index.html` — the entire single-page site. All sections (Hero, Explore, Travel Map, Tiny Discoveries, Essays, Gallery, Projects, About Me, Contact) live in one file, navigated via in-page anchor links (`#explore`, `#map`, `#essays`, etc.) rather than separate routes.
- `style.css` — one stylesheet for the whole site, including both the homepage design system and the longform essay layout (see below). CSS custom properties in `:root` (`--bg`, `--ink`, `--accent`, etc.) define the palette; change them there rather than hardcoding colors elsewhere.
- `script.js` — all homepage interactivity, in four independent, unrelated pieces:
  1. Scroll-position memory (saves/restores exact `scrollY` per-path via `sessionStorage`, with `history.scrollRestoration = 'manual'`).
  2. Hero → Explore transition (the "Begin" button locks hover states via a pointermove listener until the user actually moves the mouse, to avoid accidental hover-triggered animations after the scroll).
  3. `IntersectionObserver`-driven scroll reveals for the Explore intro text and list items.
  4. Travel map pin interactions — clicking a `.pin` button rewrites `#mapNote`'s innerHTML from the `notes` object keyed by place name.
- `cursor.js` — the site-wide custom cursor (see "Custom cursor & corner plant" below). Included on all 19 HTML pages via `<script src="cursor.js">` (`../cursor.js` from subdirectories).
- `corner-plant.js` — the homepage-only Easter egg (see below). Included only in `index.html`.
- `essays/*.html` — standalone longform essay pages, each a full HTML document (own `<head>`, own Google Fonts `<link>`s) that links back to `../style.css` and `../images/`. They share one structural template: `essay-page` body class → `.reading-progress` bar → `.article-header` (title/subtitle/meta) → `.article-body` (drop cap, pull quotes, `article-bridge`/`article-ending` styled asides) → `.article-references` → `.article-footer`.
  - **Known duplication**: each essay page inlines its own `<script>` for the reading-progress bar calculation — identical logic copy-pasted three times rather than factored into `script.js`. When editing this behavior, update all three essay files.
  - When adding a new essay, copy the structure of an existing one in `essays/` (matching class names exactly, since all styling comes from the shared `.article-*` rules in `style.css`) and add a corresponding `.essay-spine` entry to the Essays section of `index.html`.
- Field Notes (`#discoveries` in `index.html`, styled by `.field-note-*` rules in `style.css`) — a horizontally scrollable row of small note cards, deliberately not styled like the Essays shelf. Each `.field-note-card` shows only a number/title/tag(s)/date by default; hovering (or `:focus-visible`) crossfades the title into a one-line hook sentence plus a reveal link (`N min →`). The row contains only real notes Elvira has sent — there are no placeholder/demo cards, and none should be added. See "Field Notes maintenance rules" below — follow this process automatically whenever Elvira sends a new note, without being asked.
- Field Note reading pages (`field-notes/*.html`, e.g. `field-notes/the-shape-of-home.html`) — deliberately **not** a copy of the essay template. Body class is `field-note-page` (not `essay-page`), nav wrapper is `field-note-page-nav`. They reuse the shared longform primitives (`.article-main`, `.longform-essay`, `.article-body`, `.drop-cap`, `.article-bridge`, `.article-ending`, `.article-footer`, same width/palette/grain/nav) but the header is intentionally shorter and quieter — see "Field Note page template" below. No reading-progress bar, no `.article-references` section (these are personal notes, not research essays).
- Gallery (`#gallery` in `index.html`, plus `gallery/photographs.html`, `gallery/drawings.html` and `gallery/concerts.html`) — a museum-style entrance, not a portfolio grid. See "Gallery architecture" below for the full module structure, data convention, and how to add real images.

## Field Note page template

This is the standing structure for every `field-notes/*.html` page — follow it exactly for new notes, don't drift back toward the essay template:

- `<body class="field-note-page">`, header `<header class="nav field-note-page-nav">` (same nav links, just pointing `Field Notes` at `#discoveries` instead of `Essays`).
- **No** `.reading-progress` bar markup, and **no** inline reading-progress `<script>` at the bottom (essays have both; field notes have neither).
- `.article-header` contains only: the `.article-back` link ("← Back to field notes"), a plain `<h1>` (no `.section-label` eyebrow above it, no `.article-subtitle` below it), and one `.article-meta` line.
- **Title**: plain text, no `<em>` emphasis span and no forced `<br>` line break — don't invent an italicized word just to echo the essay layout. Styled smaller/lighter via `.field-note-page .article-header h1` (~32–48px vs. essays' ~56–91px).
- **Meta line** (reuses `.article-meta`/`.article-meta-dot`, same dot-separated style as essays, different content): exactly four items, in this order — `No. 0XX` (the note's current number) → topic tag(s) → exact date → reading time. No author name, no visible word count.
- **Date format on the page** is the full exact date, e.g. `12 July 2026` (`<time datetime="2026-07-12">`) — this is deliberately more precise than the card's abbreviated `Jul 2026`; the two formats coexist by design (card = compact list view, page = precise record).
- Body content below the header is unchanged from the essay system: drop cap on the lead paragraph, `.article-bridge` on short standalone lines (in place, not duplicated), `.article-ending` on the closing line. No `.article-pullquote` duplication of her own sentences — if a line deserves emphasis, style it in place rather than repeating it.

## Gallery architecture

The Gallery is three completely independent modules — Photographs, Drawings and Concerts — reached through a shared entrance. It is deliberately not a portfolio: no cards, no colour, no Pinterest-style grid. Every image gets room to breathe. All three modules share one automated discovery/optimisation/generation pipeline (see "Adding real images is automated" below) — Drawings and Concerts differ from Photographs only in their content and structure (grid mode, and whether there are subcategories), never in behaviour.

**Files:**
- `index.html` `#gallery` — the entrance. Three full-bleed panels (`.gallery-panel`, in a 3-column grid, wrapping to 2+1 on tablet and 1 column on mobile), each a single `<a>` linking to one of the three exhibition pages. No buttons; the image itself is the link.
- `gallery/photographs.html` — standalone exhibition page, **room-switcher** interaction (see below): one collection visible at a time — Stillness, Light, Presence, Landscapes, in that order — switched horizontally, browsed vertically. Grid mode `"masonry"`: CSS `column-count`, cropped to fixed aspect keywords, meant to read as an organic photo stream.
- `gallery/drawings.html` — standalone exhibition page, **same room-switcher**, same nav/transition/keyboard/touch/lightbox behaviour as Photographs — Sketches, Illustrations, Comics, Graphic Designs, in that order. Grid mode `"editorial"`: CSS Grid, each artwork at its own natural `aspectRatio` (no cropping), matted tiles (`.exhibit-tile--editorial`) so it reads as composed, individual works rather than a photo stream. Photographs and Drawings share every structural and interaction layer; only the grid mode and the data differ — see "Room switcher" below for what's shared vs. what's mode-specific.
- `gallery/concerts.html` — standalone exhibition page, **no room switcher** — Concerts has no subcategories, so every concert photo lives in one collection (folder `images/gallery/concerts/concerts/` — same one-collection-per-subfolder convention as Photographs/Drawings, just with a single collection) shown directly via `TowardsWonderGallery.renderCollections()` (mode `"masonry"`, same as Photographs). Title ("Concerts.") and subtitle ("Moments carried by music.") live once in the page's own `.exhibit-header` (`.exhibit-subtitle`, a style already used elsewhere for italic sub-copy); the collection's own `title`/`sentence` are left as empty strings in the data so `gallery.js` skips rendering a duplicate intro block above the grid (see `buildCollectionSection()` — it only builds an intro when `collection.title` is truthy).
- `gallery/data/photographs-data.js`, `gallery/data/drawings-data.js` and `gallery/data/concerts-data.js` — the content, one file per module, all generated (see below). `photographs-data.js` still contains an `editors-selection` and a `details` collection — they're just not part of the room rotation (`inRotation: false`, see below).
- `gallery/gallery.js` — shared rendering engine used by all three pages (see below).
- `gallery/rooms.js` — the room-switcher, shared by Photographs and Drawings (mode-agnostic — see below). Concerts doesn't use it (no subcategories to switch between).

**Data convention — JSON-shaped, but loaded as a script, not `fetch()`.** The data files assign to `window.PHOTOGRAPHS_DATA` / `window.DRAWINGS_DATA` / `window.CONCERTS_DATA` instead of being `.json` files fetched at runtime. This is deliberate: `fetch()` of a local file fails under `file://` (CORS), which would break this site's core convention of previewing by double-clicking `index.html` with no server. Loading data as a plain `<script src="data/...">` keeps that working. If a real build step is ever introduced, converting these to true `.json` + `fetch()` is a small, mechanical change — the shape stays identical.

**Photographs schema** (`aspect` keyword, masonry mode):
```js
window.PHOTOGRAPHS_DATA = {
  totalCount: 257,               // shown on the entrance panel only
  collections: [
    {
      id: "stillness",           // used for DOM ids and lightbox state
      title: "Stillness",
      sentence: "One line, shown under the title before the grid.",
      featured: true,             // optional — bigger masonry columns if a collection using this is ever rendered in the vertical-stack style (renderCollections)
      comingSoon: true,           // optional — renders "More to come." instead of a grid
      images: [
        { id: "stillness-001", src: null, alt: "", aspect: "portrait" }
      ]
    }
  ]
};
```
`aspect` is one of `portrait` / `square` / `landscape` / `tall` / `wide` — varied, for masonry rhythm, mapped to a fixed CSS `aspect-ratio` on `.exhibit-tile`.

**Drawings schema** (`aspectRatio` value, editorial mode) — see the full field-by-field comment at the top of `gallery/data/drawings-data.js`:
```js
window.DRAWINGS_DATA = {
  totalCount: 36,
  collections: [
    {
      id: "comics",
      title: "Comics",
      sentence: "Small stories told in sequence.",
      images: [
        {
          id: "comics-morning-walk-1",
          src: null, thumbnail: null, alt: "",
          category: "comics",              // sketches | illustrations | comics | graphic-designs
          aspectRatio: "3 / 2",             // CSS aspect-ratio value — natural proportion, never forced
          featured: false,                  // optional — spans 2 grid columns
          sequence: { group: "morning-walk", page: 1 }, // optional — comic pages that belong together
          title: undefined, year: undefined, medium: undefined, // optional — surfaced in the lightbox meta line when present
          dimensions: undefined, description: undefined, order: 1
        }
      ]
    }
  ]
};
```
Every field beyond `id`/`src`/`alt`/`aspectRatio`/`category` is optional; today's placeholder data deliberately leaves `title`/`year`/`medium`/`dimensions`/`description` unset rather than inventing real-sounding artwork info — fill them in alongside `src` when Elvira supplies real drawings. `category` drives the medium-specific grid rhythm: `comics` gets a wide 2-column card (`.exhibit-tile--comics`) so panels stay legible without cropping; the others render at whatever `aspectRatio` and `featured` specify. `sequence` and `order` are structural metadata only — nothing groups sequence pages into a sub-carousel or resorts by `order` yet; both are there for whenever that's worth building.

**Adding real images to any of the three modules is automated — do not hand-edit `photographs-data.js`, `drawings-data.js` or `concerts-data.js`.** All three are generated files (see the banner at each one's own top). The workflow is identical across modules:
1. Drop new images into `images/gallery/<module>/<folder>/` — for Concerts that's always the single `images/gallery/concerts/concerts/` folder (see the `gallery/concerts.html` bullet above); for Photographs/Drawings, whichever collection folder they belong to — named sequentially (`presence86.jpeg`, `presence87.jpeg`, ...) — or a whole new folder for a brand-new collection.
2. Run `npm run prepare-gallery` (or the two steps separately — `npm run optimise-gallery` then `npm run generate-gallery`).
3. Commit the result.

- **`scripts/optimise-gallery-images.js`** — recursively optimises every file in `images/gallery/{photographs,drawings,concerts}/` only (never the logo, cursor assets, hero image, or anything else). Converts HEIC/HEIF to WebP; re-encodes JPG/PNG/WebP at a size-aware quality (targeting ~300–600KB) after resizing anything over `MAX_LONG_EDGE_PX` (3000px) — never upscales, never replaces a file with a larger one. Every file about to be modified is backed up first to `gallery-originals-backup/` (gitignored, since `images/gallery/` itself isn't tracked in git — this is the only recovery path for what is otherwise a lossy, irreversible edit), under a manifest key namespaced by module (`photographs/stillness/…`, `drawings/sketches/…`, `concerts/…`) so identically-named files in different modules can never collide. A manifest in that same folder records each file's post-optimisation (size, mtime) signature so a file untouched since the last run is skipped instantly, not re-compressed again (repeated lossy re-encoding would quietly degrade quality for no reason) — a one-time migration on load upgrades any manifest written before Drawings/Concerts existed (plain, un-namespaced keys, implicitly all Photographs) to the namespaced form, so this generalisation never triggered a wasted re-compression pass over the 192 already-optimised photographs. All tunables (quality tiers, max dimension, size targets) are `EDIT POINT`-commented at the top of the file.
- **`scripts/generate-gallery-data.js`** — a `MODULES` config array (one entry per module: Photographs, Drawings, Concerts) drives one shared pipeline that scans `images/gallery/<module>/` and rewrites that module's `gallery/data/<module>-data.js` from scratch. Collection copy (`title`/`sentence`/`featured`/placeholder count, plus per-collection `aspects`/`featuredIndexes` for Drawings' editorial placeholders) lives in each module's `collectionsMeta` table inside this script, **not** in the generated files — that's the one place to edit wording. Each `collectionsMeta` entry names the folder it expects (`folder:`), which doesn't have to match the collection `id` — e.g. Photographs' `landscapes` collection folder on disk is the singular `landscape/`, and three of Drawings' four folders are similarly singular/differently-spelled (`illustration/`, `comic/`, `graphic design/` — only `sketches/` matches its id exactly). A collection with no matching folder yet keeps rendering its usual `placeholderCount` numbered tiles, exactly as before. Files within a folder are naturally sorted (`presence2` before `presence10` before `presence100`, via `scripts/lib/gallery-fs.js`'s `naturalCompare`), and each real image's `aspectRatio` is read directly from the file with `sharp(...).metadata()` (EXIF-orientation-aware) rather than guessed — see "Real photos keep their true aspect ratio" below. A folder on disk with no matching `collectionsMeta` entry is still included (nothing is ever silently dropped) but flagged loudly in the script's console output, since a genuinely new collection also needs a title/sentence written by hand and adding to that module's room order — both deliberately left as manual, curatorial steps. Every module's config also supports `flat: true`, which would tell the shared pipeline to read images directly from `images/gallery/<module>/` instead of per-collection subfolders and skip room-order/unclaimed-folder logic — not currently used by any module (Concerts turned out to still want a single named subfolder, matching the other two modules' convention), but available if a future module ever needs a genuinely flat layout.
- **`inRotation` flag** — each `collectionsMeta` entry (Photographs, Drawings) may set `inRotation: false` to keep a collection's data generated (still in that module's data file, never silently dropped) but excluded from the two things actually shown to visitors. Currently only Photographs uses this, to exclude `editors-selection` and `details`. The script derives two things from it on every run, per module:
  - **`gallery/data/<module>-rooms.js`** — a small generated file (loaded by that module's page before its `-data.js`) that sets `window.<MODULE>_ROOM_ORDER` to `collectionsMeta.filter(m => m.inRotation !== false).map(m => m.id)`. This is the single source of truth for which collections are switchable rooms and in what order — `photographs.html`/`drawings.html` pass it straight into `initRooms()` rather than hardcoding an array, and `gallery-preload.js` (below) also reads it to know which collection is "first" for its idle-time warm-up. Bringing a collection back into rotation (e.g. reviving Editor's Selection or Details) is a one-line change: flip its `inRotation` to `true` (or remove the flag) and re-run `npm run generate-gallery` — no hand-edits needed to the page or anywhere else. Concerts has no rooms file (`outputRoomsPath: null` in its module config) since it has no room switcher.
  - **Live entrance-panel count** — the script also computes, per module, the total image count (real-photo count where real, placeholder count where not) across only that module's in-rotation collections, and patches it directly into that module's own entrance panel on `index.html` (`.gallery-panel-count`, scoped by regex to each panel's own class so the other two panels' counts are never touched). This means all three homepage counts are regenerated, not hand-maintained — always matching what's actually in each gallery after any `npm run generate-gallery` run. (Regenerating this way also fixed a pre-existing drift: the Drawings panel had been hand-typed as "48 works" without ever matching the actual data — it's now always correct.)
- **`layoutSort` flag** — a `collectionsMeta` entry may set `layoutSort: true` to have its real images reordered by measured aspect ratio (via `reorderForLayoutRhythm()`) instead of kept in filename order. Currently only Drawings' `illustrations` uses this — Elvira asked for that one category specifically to be free to shuffle for a better-looking layout, unlike the others (Sketches keeps upload order; Comics' filename order encodes a real sequence; Graphic Designs is small enough that order barely matters). This exists because `.exhibit-grid--editorial` is a plain 3-column CSS grid with `align-items: start`, not a masonry layout — a row's height is set by its tallest item, and shorter row-mates just leave dead space beneath them rather than reflowing around it. The function sorts images by height-per-width, chunks them into complete rows of 3 (any remainder is set aside and appended at the very end, so it can't misalign every row after it), then interleaves the sorted rows short/tall/short/tall rather than leaving them a straight short-to-tall ramp — so the grid reads as varied rather than mechanically graduated, while each row's own three images stay close in height. Deterministic and safe to run on every regenerate: the same images always produce the same order, and a newly added photo only nudges the couple of rows near its own height.
- **`scripts/watch-gallery-images.js`** — optional dev convenience, not required for the pipeline to work. Watches `images/gallery/` (`fs.watch(..., { recursive: true })`) and automatically re-runs `optimise-gallery-images.js` + `generate-gallery-data.js` whenever a file changes there, debounced ~1.5s so a batch of dragged-in files triggers one run rather than one per file. Run via `npm run watch-gallery`; leave it running in a terminal while adding real photos/drawings/concert shots and the local data files + homepage counts stay in sync without having to remember to run `npm run prepare-gallery` by hand. This only keeps a *local* preview in sync — the deployed site still needs a commit + redeploy afterward, since Towards Wonder has no server/build step to auto-publish from. (The optimiser rewriting an image file in place is itself a filesystem change inside the watched tree, so it can trigger one harmless extra pipeline pass after a real run — the manifest's unchanged-since-last-run check makes that pass a fast no-op, not a second real recompression.)
- Both scripts are dev-only tooling declared in `package.json` (`sharp` as the only dependency) — they don't change the site's own zero-build-step static deployment. `npm run prebuild` chains to `prepare-gallery`, for hosts that run a build step; this project's current Netlify-Drop/GitHub-Pages deploy path (see "Commands" above) doesn't, so the two npm scripts need to be run by hand before committing/deploying today.

**`gallery.js` responsibilities** (the shared engine all three pages build on):
- `TowardsWonderGallery.renderCollections(data, rootSelector, mode)` — vertical stack of every collection in `data`, appended into one root. Used by `concerts.html`, which has a single collection and no room switcher; Photographs and Drawings still use the room switcher instead.
- `TowardsWonderGallery.buildCollectionSection(collection, mode)` — builds one collection (intro + grid + infinite scroll) and returns it **detached**, for a caller to place wherever it wants. `mode` is `"masonry"` or `"editorial"` — see `createTile()` for how each mode sets tile classes/aspect-ratio. `rooms.js` calls this once per room panel on Photographs/Drawings; `renderCollections()` calls it once for Concerts. Skips building the intro block (title/sentence, and the editorial work-count line) entirely when `collection.title` is falsy — Concerts' single collection has an empty title/sentence for exactly this reason, since its title/subtitle already appear once in the page's own `.exhibit-header`. Every other collection on the site always has a non-empty title, so this is a no-op for Photographs and Drawings.
- Optional artwork count line (`.exhibit-collection-count`, e.g. "10 works") under the collection sentence — only rendered when `mode === "editorial"`, so the Photographs header is untouched.
- Batches images in 12 at a time per collection (infinite scroll), via one `IntersectionObserver` per collection watching a trailing sentinel element — not a single global scroll listener.
- Native `loading="lazy"` on real `<img>` tags for lazy loading once real `src` values exist.
- A single shared fullscreen lightbox (built once, reused across every tile on the page, regardless of which collection or room it's in): fade + gentle scale transition, ←/→/Esc keyboard navigation, click-outside-to-close, focus returns to the originating tile on close. `TowardsWonderGallery.isLightboxOpen()` is exposed so other code (the room switcher) can check before acting on the same keys. An optional metadata line (`.lightbox-meta`, title · year · medium, whichever parts exist) sits above the index/total caption and is hidden via `:empty` when an image carries none of those fields — true for every Photographs image and every placeholder Drawings image today, so it's invisible until real artwork metadata is added.
- A separate `IntersectionObserver` fades/rises each tile in as it enters the viewport (respects `prefers-reduced-motion`).

**Real photos keep their true aspect ratio, not a cropped keyword.** Every real image object in `photographs-data.js` (as written by `generate-gallery-data.js`) carries an `aspectRatio: "<width> / <height>"` field alongside the legacy `aspect` keyword. In `gallery.js` `createTile()`, masonry-mode tiles still get the cycled `aspect` keyword class as before (`exhibit-tile--portrait` etc.) as a fallback, but when `image.aspectRatio` is set it's applied as an inline style — which wins over the keyword class — and the tile also gets `.exhibit-tile--natural`, which switches its `<img>` to `object-fit: contain` (style.css). Since the tile's own ratio now exactly equals the photo's, `contain` and the default `cover` render pixel-identically with zero letterboxing — it's there as a guarantee against ever cropping real content, not a visual change. Placeholder collections have no `aspectRatio` field and are completely unaffected — same keyword-cycled masonry rhythm and `cover` cropping as before. This is why the CSS multi-column masonry (`column-count`) is enough on its own to produce an organic, staggered layout once real (varied) ratios flow through it — no separate positioning system was needed.

**Room switcher (`gallery/rooms.js`, shared by Photographs and Drawings — Concerts has no room switcher)**: turns a list of collections into "exhibition rooms" — only one in the DOM flow at a time, switched by click / ←→ arrow keys / touch swipe / trackpad horizontal wheel, never by page reload, and cyclically (there is always a previous and next room, wrapping past the first/last). `initRooms(data, roomIds, navSelector, stageSelector, mode)` takes the grid mode as its last argument (`"masonry"` for `photographs.html`, `"editorial"` for `drawings.html`, defaulting to `"masonry"` if omitted) and passes it straight through to `buildCollectionSection()` — `rooms.js` itself has no opinion on how a room's grid looks, only on which room is showing. Mechanically:
- All rooms are built up front via `buildCollectionSection()` (this is the "preload adjacent collections" Elvira asked for — with a handful of placeholder rooms, building all of them up front is simpler than true adjacent-only lazy building and costs nothing noticeable; revisit only if real content ever makes a single room's initial batch expensive).
- **Nav design**: `.room-nav` shows only three items — previous / current / next room title (e.g. "← Light   Stillness   Presence →"), not all five at once. Previous/next are muted with a small arrow that nudges on hover; current is bolder/accent-coloured. The three labels are rebuilt (not just re-highlighted) on every room change, since which collections count as "prev"/"next" changes too — see `updateNav()` in `rooms.js`. This was chosen over showing all five names at once (which Elvira had explicitly allowed as a fallback) because it reads calmer and more like standing between two doorways than clicking a tab bar. `.room-nav` stays `position: sticky` under the site's top nav.
- **Cyclic navigation**: `cycle(index, delta) = (index + delta + rooms.length) % rooms.length` — going next from the last room wraps to the first, and vice versa, so the nav never has to grey out an unusable arrow. `goToRoom()` takes an explicit `direction` parameter from `goNext()`/`goPrev()` (rather than inferring it from the raw index difference) so the slide direction is still correct at the wrap boundary.
- **Edge fades** (`.room-stage-edge--left/right`): two fixed-position gradient strips at the page edges, present at all times (navigation is cyclic, so the exhibition always continues past either side) — a quiet, wordless hint that horizontal interaction exists, deliberately not tutorial text like "swipe left or right".
- Switching rooms: the outgoing and incoming panels are both set `position: absolute` and the stage's `min-height` is locked to the outgoing panel's height first, so nothing jumps; the incoming panel starts translated off to one side (`.is-entering-from-left/right`), a forced reflow (`void el.offsetWidth`) makes that starting position "stick" for one frame, then both panels' classes flip to their resting state so the CSS `transition` actually animates between the two — a fade + short horizontal slide, direction-aware. After the transition timeout, the outgoing panel is fully hidden and the incoming one returns to normal document flow.
- Arrow-key nav is gated on `!TowardsWonderGallery.isLightboxOpen()` so it never fights the lightbox's own ←/→ image navigation.
- Touch swipe requires the horizontal delta to clearly dominate the vertical one before it acts, and the stage has `touch-action: pan-y`, so vertical scrolling to browse photographs is never intercepted — the two interactions (browse vs. switch room) are structurally kept apart, not just tuned to feel separate. Once a gesture reveals itself as horizontal, its `touchmove` calls `preventDefault()` (via a `{ passive: false }` listener) for the rest of that gesture — otherwise the same horizontal drag is also read by the browser itself as an edge-swipe "go back" gesture, which would leave the Photographs page entirely and land back on the Hero section. A gesture that reveals itself as vertical is never touched.
- On mobile (`max-width: 760px`), the prev/next room *names* are hidden and only their arrows remain (plus the current room's name) — showing all three full titles at once got tight on narrow screens; the arrows alone are enough to signal "more rooms this way."

**Visual system**: scoped to `.gallery-page` / `.gallery-entrance` — background `#f8f7f4` (Elvira's spec, distinct from the homepage's `--bg`), same typography and grain as the rest of the site. Motion is intentionally slow (0.5–1.2s, calm easing) and small (hover scale caps at 1.03) — do not add shadows, dashboard-style cards, boxed/coloured tabs, or fast/dramatic transitions here; that would work against the "quiet museum" brief this system was built to.

**Background gallery preload (`gallery-preload.js`)**: warms the browser's image cache for all three Gallery modules *before* a visitor opens one, so each feels instant on arrival, without adding any cost to the page it runs on. Included on `index.html` (`<script src="gallery-preload.js">`) and every `essays/*.html` page (`<script src="../gallery-preload.js">`) — deliberately not on the gallery pages themselves (nothing to warm up for a page already open) or on Field Notes (not part of this feature's brief). It never builds gallery markup, never runs `gallery.js`'s reveal/hover animations, and never touches the DOM beyond a couple of dynamically-inserted `<script>` tags per module, used to fetch that module's own data files.
- **`GALLERIES` config array** — one entry per module (Photographs, Drawings, Concerts): its data script paths, `dataVar`/`roomsVar` globals, and the link selector that counts as "heading toward this gallery" (`a[href*="photographs.html"], a[href$="#gallery"]` for Photographs — the homepage's generic back-to-gallery anchor defaults to it — `a[href*="drawings.html"]` for Drawings, `a[href*="concerts.html"]` for Concerts). Concerts' `roomsVar` is `null` since it has no rooms file; its "first collection" is just `data.collections[0]` instead of the first id in a room order.
- **Two-stage warm-up, per module**: on `requestIdleCallback` (falls back to a `setTimeout(…, 2000)` on Safari, which has no `requestIdleCallback`), each module loads its rooms file (if any) + data file via dynamically-created `<script>` tags (so they cost nothing during normal page load — only once genuinely idle does any network request happen) and warms its first collection's first 12 images (`IDLE_BATCH_SIZE`, matching `gallery.js`'s own batch size) with off-DOM `new Image()` objects. Deliberately not `<link rel="preload">`, which signals elevated fetch priority — the opposite of a quiet background warm-up. On `mouseenter`/`focus`/`touchstart` of a module's own link, it immediately warms a further 12 (`HOVER_BATCH_SIZE`) for that module only, so more of that specific exhibition is cache-warm by the time a click actually lands. A single shared `preloadedUrls` map (keyed by resolved URL) prevents ever re-requesting the same image across modules or stages. Drawings and Concerts have no real images yet (every tile's `src` is `null`), so warming them today is a genuine no-op — zero network requests — until real content exists, at which point it starts warming automatically with no changes needed here.
- **Cross-page-depth URL resolution**: image paths in each data file are written relative to `gallery/*.html` (e.g. `"../images/gallery/photographs/presence/presence1.jpeg"`), which only resolves correctly from within `gallery/`. `gallery-preload.js` instead resolves every path against its own script location (`document.currentScript.src` → project root, via `new URL(".", scriptEl.src)`), which is correct whether the *page* running it is at the root (`index.html`) or one level down (`essays/*.html`) — confirmed in testing: the same script fetches the right absolute URLs from both.
- **Guards**: skips entirely if `navigator.connection.saveData` is set, or `effectiveType` is `slow-2g`/`2g`. Proceeds if the Network Information API is absent (Safari/Firefox), since there's no signal either way.
- `IDLE_BATCH_SIZE`, `HOVER_BATCH_SIZE`, and `GALLERY_LINK_SELECTOR` are called out as `EDIT POINT`s at the top of the file.

## Custom cursor & corner plant

Two isolated, opt-in-only features layered on top of the whole site — neither one touches any existing markup, section, or script. Both are fine-pointer/hover-capable-device features; touch devices get the native cursor and never run the plant's proximity logic.

**Files:**
- `cursor.js` — builds and drives the custom cursor. Included on all 19 HTML pages (`<script src="cursor.js">` from `index.html`, `<script src="../cursor.js">` from every subpage). Does nothing (returns immediately) unless `window.matchMedia("(pointer: fine) and (hover: hover)")` matches.
- `corner-plant.js` — builds and drives the corner-plant Easter egg. Included only in `index.html`. Cooperates optionally with `cursor.js` via `window.TowardsWonderCursor.setSeedMode()` but degrades silently if that global isn't present (e.g. cursor disabled on this device).
- CSS for both lives in `style.css`, in two clearly delineated sections at the end of the file ("CUSTOM CURSOR" and "EASTER EGG — CORNER PLANT"), following the site's one-stylesheet convention. All tunable values (sizes, colours, opacity, easing curves, interaction radius, dwell time, messages) are called out with `EDIT POINT` comments in both the CSS and the JS.

**Cursor mechanics**: two-layer DOM structure — an outer `.custom-cursor` wrapper that JS repositions every animation frame via `translate3d` (rAF + linear interpolation for the trailing-delay feel; no CSS transition on position, since the lerp itself is the smoothing), and an inner `.custom-cursor-dot` whose scale/opacity/border-radius are controlled purely by CSS classes (`.is-hover`, `.is-pressed`, `.is-seed`) so their transitions animate independently without fighting the per-frame position updates. Hover detection is delegated (`pointerover`/`pointerout` on `document`, matched against a generic tag/role selector — `a, button, input, textarea, select, summary, label, [role="button"], [tabindex]`) rather than tied to page-specific class names, so the same file works unmodified on every page, including content added later (gallery infinite-scroll tiles, the lightbox). `html.custom-cursor-active { cursor: none !important }` (added by JS only after the pointer/hover check passes) hides the native cursor everywhere except text inputs, which keep a normal caret.

**Restrained mode**: `essay-page` and `field-note-page` bodies get the `.custom-cursor--restrained` modifier (smaller hover size, lower opacity) so the cursor never competes with long-form reading — a deliberate interpretation of Elvira's "make it especially restrained on essay pages" note, extended to Field Notes since they're the same kind of long-form reading page.

**Corner plant mechanics** (`index.html` only): a small single-line SVG plant, `position: fixed` bottom-right, built entirely by `corner-plant.js` (nothing hand-written in `index.html`). The stem and both leaves are deliberately not straight/mirrored — a slight bow in the stem and two differently-sized, differently-curved leaves, so it reads as hand-drawn rather than geometric; both ends of the stem stay pinned to the same coordinates the rest of the system depends on ((32, 84) base, (32, 24) top — the stem's own apical tip).

A `pointermove` listener throttled to one check per animation frame computes distance from the cursor to the plant's leaf centre; inside `NEAR_RADIUS_PX` (150px default) the leaves tilt toward the cursor via inline `transform: rotate()` (CSS transition smooths each update) and a dwell timer starts. If the cursor stays inside the radius continuously for `GROW_DWELL_MS` (1800ms default), a small bud grows in once (`.plant-bud.is-grown`) — it does not regrow or reset on subsequent visits within the same page load.

**The bud** (`.plant-bud`, grown state only) is two concentric SVG `<circle>`s, not a leaf shape: `.plant-bud-dot` (the visible growth point, ~4.4px diameter, filled with `var(--accent)`) and a larger `.plant-bud-halo` sharing its centre, invisible at rest. Both circles are positioned so `cy + r = 24` — exactly the stem path's own top endpoint — so the stem visually terminates at the dot's centre-bottom rather than beside it; the `<g class="plant-bud">` wrapper's `transform-origin` is pinned to that same point via `transform-box: view-box; transform-origin: 32px 24px` (explicit coordinates rather than the `.plant-leaf`-style `fill-box`/`bottom center` keyword, since two differently-sized concentric circles would otherwise shift the origin depending on the halo's radius).

**Bud glow**, once grown: deliberately reuses the *same* `NEAR_RADIUS_PX`/`strength` proximity the leaf tilt above already computes — no separate tighter radius around the bud itself — so the cursor only needs to be generally near the plant, never precisely over a 4–5px target. Entering the radius sets `budHalo`/`budDot`'s `transitionDuration` to the shorter `BUD_GLOW_ENTER_MS` (600ms) and then, every frame while near, sets the halo's `opacity` to `BUD_GLOW_MAX_OPACITY * strength` (max 0.22) *and* its own `transform: scale()` between `BUD_GLOW_MIN_SCALE`/`BUD_GLOW_MAX_SCALE` (0.6 → 1.5) — so "near" and "very near" read as visibly different amounts of glow, not just a dimmer/brighter version of the same size — while the dot only gets a slight `opacity` brighten (`BUD_REST_OPACITY + BUD_BRIGHTEN_MAX * strength`, 0.85 → 0.97) and never scales, so the bud itself never reads as enlarging. Leaving the radius switches to the longer `BUD_GLOW_EXIT_MS` (850ms) before dropping both back to rest (halo scale resets to `BUD_GLOW_MIN_SCALE`). `.plant-bud-halo` scales from its own centre via `transform-box: fill-box; transform-origin: center` (its bbox center coincides with the dot's center by construction, both circles sharing the same `cx`/`cy`). The halo's blur itself (`filter: blur(var(--bud-glow-blur))`, 7px) is a static value, not JS-animated — a plain blurred fill circle rather than `filter: drop-shadow(...)` with colour/blur driven by CSS custom properties, which an earlier version tried: it was visually correct (confirmed by screenshot) but its computed `filter` value couldn't be reliably read back via `getComputedStyle` in this repo's testing tool (see the caveat below), so the blurred-circle approach is both simpler to verify and renders identically across Safari/Chrome.

**Hover message**: shown automatically on `pointerenter`/`pointerleave` of the `.corner-plant` container (not `:hover`, not click) — no click needed, and clicking does nothing. The four messages cycle in a fixed order (not randomly), advancing only once a hover has fully finished: pointer leaves → `EXIT_DELAY_MS` (300ms) grace period, so a brief accidental leave-and-return doesn't count → fade-out (`EXIT_FADE_MS`, 850ms, duration set via inline style since it differs from the CSS default entrance duration) → only then does `messageIndex` advance. Re-entering at any point during the delay or the fade cancels both timers and re-shows the *same* message rather than advancing. `.corner-plant-message` becomes `pointer-events: auto` only while `.is-visible`, so once shown it's part of the container's own hit region — moving from the plant up onto the message (they're both descendants of `.corner-plant`, and `pointerenter`/`pointerleave` fire only for the container's whole hit region, not internal boundaries) doesn't trigger a leave. Both the tilt and the growth animation, and the exit-duration override, are skipped under `prefers-reduced-motion` — the stylesheet's own reduced-motion rule (near-instant transitions) governs instead.

**Known testing-environment caveats**:
- This repo's headless browser testing tool backgrounds its tabs by default (`document.visibilityState` reports `"hidden"`), which fully suspends `requestAnimationFrame` — so the plant's proximity/tilt/growth/glow logic (all rAF-gated) cannot be exercised end-to-end via that tool even though the underlying distance math and DOM logic are straightforward and correct. Verify those specific behaviours with a real cursor in a real (non-backgrounded) browser tab. The hover-message system is unaffected by this (it's driven by `pointerenter`/`pointerleave` + `setTimeout`, not rAF) and was confirmed working end-to-end — sequence order, cancel-on-re-entry, exit delay/fade timing — via direct event-dispatch assertions; all of `cursor.js`'s hover/press states were confirmed the same way.
- Separately, `getComputedStyle()` in this same tool does not reliably reflect a property's value while a CSS `transition` on that property is active or has just been triggered — confirmed across `transform`, `filter`, and `opacity` (i.e. it isn't specific to any one property). Setting `transition: none` immediately before reading resolves it every time, and a real screenshot always shows the true, correct rendering regardless. When verifying anything transitioning in this tool, prefer a screenshot or a `transition: none` read over trusting `getComputedStyle` directly.

## Field Notes maintenance rules

Elvira will send real Field Notes over time — full article text plus a publication date. For every note she sends, follow this process automatically, without being asked:

1. **Preserve the article.** Use her title and article text as given; don't rewrite, shorten, or alter the argument unless she explicitly asks for an edit. Create (or update) the note's full reading page in `field-notes/` following the "Field Note page template" above, and point the card's `href` at it.
2. **Reading time.** Word count ÷ 220 wpm, rounded **up** to the nearest whole minute, minimum 1. Replace the card's `Read →` with `N min →` (exactly that format — not "min read"). On the reading page itself, the same figure is shown as `N min read` in the meta line. Recalculate whenever the article text changes.
3. **Topic tag.** Pick the single most fitting tag from: Humanity, Meaning, Home, Belonging, Growth, Presence, Identity, Perception, Relationships, Dentistry, Design, Travel, Nature, Technology, Care, Beauty, Art, Ethics. If Elvira specifies the topic herself when sending a note, use exactly what she says (add it to this list if new) rather than reclassifying it. Use two only when both are genuinely central (format `Tag · Tag`, matching the existing separator). Tags may be revisited/reclassified later as the collection grows — when a note is reclassified, update both the card and the reading page's meta line together (they must always match).
4. **Hover sentence** (`.field-note-hook`). Does not need to be an exact quote from the article — it can be an original "editor's line": a short, inviting sentence written for the card, in the spirit of an epigraph or pull-quote at the front of an essay collection. Either way it must be concise, capture the piece's core argument or emotional/philosophical core, and spark curiosity — never a summary, description, or clickbait. Wrap it in curly quotation marks (`“…”`) in the card markup — this applies to every hover sentence, past and future. Keep it short enough to sit comfortably in the card.
5. **Date.** Use exactly the date she provides. Card shows the abbreviated form (e.g. `Jan 2026`); the reading page shows the exact form (e.g. `12 January 2026`). Never infer or invent a more precise date than given.
6. **Reorder + renumber the whole collection.** Numbers are continuous starting at `001` = oldest; numbers increase with recency; the newest note gets the highest number. Cards run newest → oldest, left → right (highest number = leftmost `.field-note-card`). Every time a note is added, re-sort and renumber *all* cards by date — don't just append — since she may send notes out of chronological order. When a note's number changes, update its `No. 0XX` on its reading page too.
7. **Same-date ties.** If two notes share a date, preserve the order she provided them in; don't invent a more precise time to break the tie.
8. **Song lyrics.** If a note quotes a song, don't reproduce the lyric text in the published page — omit that line, leave an HTML comment noting it was skipped, and flag it to Elvira so she can add it herself or say how she'd like the moment represented instead.
9. **Preserve the existing design.** Keep the current card layout/typography/spacing/hover transition and the Field Note page template as-is; don't redesign either while adding content unless explicitly asked. Insert the new card into its correct chronological position in the row rather than just appending it at one end.

**Before implementing**, report: word count, calculated reading time, chosen tag(s), the selected hover sentence, and where the note lands chronologically relative to existing notes. Then make the changes.

## Project Philosophy

Towards Wonder is not a portfolio.

It is an editorial space for thoughtful writing, travel, and curiosity.

Design principles:

- calm over stimulation
- editorial rather than corporate
- generous whitespace
- typography first
- subtle interactions only
- Scandinavian-inspired minimalism
- every animation should feel intentional

Avoid:

- flashy UI
- gradients
- unnecessary JavaScript
- dashboard aesthetics
- decorative effects without purpose
